# 专题验收助手 PRD 与经验沉淀

更新时间：2026-07-04  
项目：campaign-review-assistant  
定位：面向网页专题、移动端活动页、小程序、游戏内嵌页、网吧内嵌页的 AI 验收工作台

## 1. 背景

专题页面上线前，产品需要同时核对设计稿、业务文案、需求/UE 描述和真实页面表现。传统做法依赖人工逐项翻资料、截图、复制文案、点按钮，容易漏掉入口、文案、跳转和背景图差异。

本工具的目标不是做一个泛泛的个人知识库，而是做一个“验收助手”：把页面 URL、设计稿截图、CMS 文案、PRD/UE、本地资料文件夹放到一个工作流里，自动整理出页面展示、页面内容、页面功能三类验收结论。

## 2. 用户与场景

### 目标用户

- 网站专题产品经理：需要在上线前快速完成活动页验收。
- 活动运营/业务方：需要确认页面展示的业务文案是否和 CMS 或需求一致。
- 前端开发/测试协作者：需要定位问题来源，判断是页面实现、配置文案、设计稿还是需求描述不清。

### 典型场景

- 移动端专题页上线前，检查右上角是否有“下载游戏”，中间是否有预约按钮，背景图是否与设计稿一致。
- CMS 后台配置了公告、按钮文案、活动资讯，需要检查页面 DOM 或截图 OCR 里是否展示一致。
- PRD/UE 写了按钮跳转、弹窗、领取逻辑，需要判断页面功能是否有入口并能点测。
- 本地资料散落在 `D:\vscode\动效\docs` 一类文件夹里，需要按项目快速搜索相关资料。

## 3. 产品目标

1. 降低专题验收启动成本：用户只需要粘贴 URL、文案、资料路径或上传截图，就能进入验收流程。
2. 缩短人工整理时间：自动输出“页面展示、页面内容、页面功能”三段结论。
3. 提高资料完整性意识：明确告诉用户缺少设计稿、CMS、PRD/UE、截图或点测证据。
4. 形成可复用报告：支持复制和导出 Markdown，方便沉淀和流转。
5. 支持渐进增强：先完成本地规则和证据链，后续再接入真正的多模态 AI 或知识库检索。

## 4. 非目标

- 不做像素级视觉验收结论。视觉 diff 只作为参考证据，不作为是否通过的最终依据。
- 不替代人工最终判断。工具输出建议，产品仍需确认上线风险。
- V1 不做多人协作、账号权限、云端项目管理。
- V1 不做完整 PRD 语义理解和复杂业务流程推理，PRD/UE 功能对比作为二期增强。
- V1 不做 Obsidian 类个人知识库，重点是“专题验收任务流”。

## 5. 当前核心流程

1. 用户输入项目资料：页面 URL、设计稿截图、CMS 文案、PRD/UE 摘要、本地资料文件夹。
2. 系统识别输入来源：告诉用户识别到了哪些资料，哪些资料缺失。
3. 用户点击开始验收。
4. 系统执行流程：识别资料 -> 采集证据 -> 对比检查 -> 生成结论。
5. 页面输出三段验收结论：页面展示、页面内容、页面功能。
6. 用户按需展开高级细节：证据、OCR、点测截图、报告预览。
7. 用户复制或导出 Markdown 报告。

## 6. 功能需求

### P0 已实现/必须保留

#### 6.1 主输入区

描述：用户可以在一个输入框内粘贴混合资料，包括 URL、CMS 文案、需求描述、本地资料路径。

验收标准：

- 输入 URL 后，系统可以识别页面地址。
- 输入 Windows 文件夹路径后，系统可以识别资料目录。
- 输入普通文本后，系统作为 CMS/需求/UE 描述候选。
- 点击“开始验收”后，不会因为输入混合内容而中断流程。

#### 6.2 资料识别预览

描述：输入区下方显示系统识别到的资料状态。

验收标准：

- 展示页面 URL、设计稿、CMS/需求、本地资料、页面截图状态。
- 已有资料显示“已填写/已上传/路径已填”。
- 缺失资料显示“待上传/待粘贴/自动采集”。

#### 6.3 一键验收流程

描述：点击开始验收后，系统按顺序执行验收任务。

验收标准：

- 流程显示四步：识别资料、采集证据、对比检查、生成结论。
- 当前步骤有 active 状态。
- 完成步骤有 done 状态。
- 失败时显示流程中断原因。

#### 6.4 页面采集

描述：根据 URL 截取移动端页面，提取 DOM 文本、图片、背景图、可点击元素。

验收标准：

- 成功采集后左侧“页面截图”状态更新。
- 页面 DOM 文本可用于 CMS 文案对比。
- 采集失败时展示明确错误。

#### 6.5 CMS 文案对比

描述：将 CMS/业务方文案与页面 DOM 文本、OCR 文本进行匹配。

验收标准：

- 输出匹配数量和疑似缺失数量。
- 文案缺失会进入页面内容结论。
- OCR 修正后可重新参与文案对比。

#### 6.6 三段验收结论

描述：默认结果区只输出三段正文，而不是复杂卡片。

验收标准：

- 页面展示：说明设计稿、页面截图、背景、入口等展示风险。
- 页面内容：说明 CMS/业务文案是否疑似缺失。
- 页面功能：说明按钮入口、跳转、弹窗是否有证据。
- 资料不足时明确提示“等待某类资料”。

#### 6.7 缺失资料建议

描述：结果区给出下一步该补什么资料。

验收标准：

- 缺 URL 时提示补 URL。
- 缺设计稿时提示上传设计稿。
- 缺 CMS 时提示粘贴 CMS 文案。
- 缺本地资料时提示搜索资料文件夹。
- 缺功能描述时提示补 PRD/UE。

#### 6.8 左侧资料栏

描述：左侧展示资料来源和状态，并支持点击查看详情。

验收标准：

- 可点击页面 URL、设计稿截图、页面截图、CMS 文案、本地资料、OCR 文案、功能点测。
- 点击后显示对应输入、预览或下一步说明。
- 不影响主结果区的简洁性。

### P1 近期优化

#### 6.9 高级细节重构

描述：把高级细节从长页面整理为 tabs。

建议分组：

- 证据：设计稿、页面截图、视觉参考。
- OCR：图片识别结果和人工修正。
- 点测：功能点击前后截图、URL 变化、失败原因。
- 报告：Markdown 预览、复制、导出。

验收标准：

- 默认不展开高级细节。
- 展开后不影响主流程阅读。
- 每个 tab 只展示当前任务相关信息。

#### 6.10 验收状态等级

描述：给最终结论增加统一状态。

建议状态：

- 可上线：无明显风险。
- 有风险可上线：存在非阻塞问题。
- 阻塞上线：存在必须修复问题。
- 资料不足：无法判断。

验收标准：

- 状态由问题数量、缺失资料、人工确认结果综合生成。
- 用户可以手动修改最终状态。
- Markdown 报告中包含该状态。

#### 6.11 模板化检查项

描述：按页面类型提供默认验收模板。

模板：

- 网页移动端专题。
- 小程序。
- 游戏内嵌页。
- 网吧内嵌页。

验收标准：

- 选择模板后自动填充建议检查项。
- 模板不会覆盖用户已输入资料。
- 模板检查项可被用户编辑或忽略。

#### 6.12 本地资料分类

描述：搜索本地资料后按类型归类。

分类：

- 设计稿/截图。
- CMS/文案。
- PRD/需求。
- UE/交互。
- 历史验收资料。

验收标准：

- 搜索结果不只是文件列表，而是按验收用途展示。
- 用户可以把某个文件设为设计稿、CMS 文案或 PRD/UE 依据。

### P2 未来方向

- 接入真正的多模态模型，理解设计稿截图和页面截图里的布局差异。
- 接入本地向量检索，让历史项目资料成为可搜索知识库。
- PRD/UE 语义对比：根据功能描述自动生成点测清单。
- 多页面批量验收：适合活动页多路由、多状态、多端对比。
- 自动生成给开发/业务方的缺陷单。

## 7. 用户故事

### 产品经理

- 作为专题产品经理，我想把 URL、设计稿、CMS 文案和资料文件夹一起丢进去，这样我不需要手动逐个整理验收依据。
- 作为专题产品经理，我想看到页面展示、页面内容、页面功能三段结论，这样我能快速判断上线风险。
- 作为专题产品经理，我想知道还缺哪些资料，这样我可以及时找业务方或设计补充。
- 作为专题产品经理，我想导出 Markdown 报告，这样我可以把验收结果发给开发、测试或业务方。

### 协作开发/测试

- 作为前端开发，我想看到功能点测失败原因和点击截图，这样我能快速定位入口、跳转或弹窗问题。
- 作为测试协作者，我想看到每个问题的证据来源，这样我能复核问题是否真实存在。

### 业务方/运营

- 作为业务方，我想确认 CMS 文案是否完整展示，这样可以避免上线后公告、按钮文案或活动资讯错误。

## 8. 成功指标

### 短期指标

- 单次专题验收准备时间减少 50%。
- 用户完成一次验收报告导出的成功率达到 80%。
- 每次验收至少识别 3 类资料来源。
- 资料缺失提示被用户采纳的比例达到 60%。

### 长期指标

- 上线前遗漏问题数量下降。
- 历史资料复用率提升。
- 专题验收报告格式逐渐统一。
- 产品、前端、业务方之间的返工沟通减少。

## 9. 风险与限制

- 页面可能有登录、地域、环境限制，导致截图失败。
- 小程序、游戏内嵌页、网吧内嵌页可能需要特殊运行环境，不能完全用普通浏览器复现。
- OCR 对设计稿文字的识别不稳定，需要人工修正。
- 视觉一致性不是像素级验收，应由产品人工最终判断。
- 功能点测依赖 PRD/UE 描述质量，描述越明确，自动点测越准确。

## 10. 开放问题

- 设计稿是否通常是单张长图，还是多张分屏截图？
- CMS 文案通常是纯文本、Excel、Word，还是后台导出的结构化文件？
- PRD/UE 最常见格式是什么：Markdown、Word、截图、HTML 原型？
- 游戏内嵌页和网吧内嵌页是否能用普通浏览器访问，还是需要模拟特定容器？
- 是否需要把验收结论同步到团队工具，例如飞书、Tapd、Jira？

## 11. 经验沉淀方法

做项目时，最容易丢失的不是代码，而是判断过程。建议每次迭代都沉淀四类东西：事实、判断、决策、复盘。

### 11.1 用一页纸记录一次迭代

每次完成一个阶段后，写 10 分钟就够。

模板：

```md
## 本次迭代

目标：
- 这次想解决什么问题？

做了什么：
- 功能 1
- 功能 2

为什么这么做：
- 当时的判断依据是什么？

遇到的问题：
- 问题 1：原因是什么，怎么解决？

用户反馈：
- 用户说了什么原话？
- 哪些反馈影响了决策？

下次优化：
- P0
- P1
- P2
```

### 11.2 把经验分成四层

#### 第一层：事实

事实是不带评价的记录。

示例：

- 用户不想做像素级视觉 diff。
- 用户希望结果是一段话，按展示、内容、功能分段。
- 用户常见资料在本地文件夹里。
- 用户希望像聊天一样丢资料。

#### 第二层：判断

判断是从事实里得出的产品理解。

示例：

- 这个工具不是个人知识库，而是专题验收工作台。
- 默认界面不能像后台，证据和高级细节应该按需展开。
- 验收结果应该帮助判断上线风险，而不是展示全部技术细节。

#### 第三层：决策

决策是你最终选择的方案。

示例：

- 默认只显示输入、识别、结果、结论。
- 结果只输出三段正文。
- OCR、点测截图、Markdown 报告放到高级细节。
- 一键验收流程显示四步状态。

#### 第四层：原则

原则是以后复用的经验。

示例：

- 产品工具先做主路径，再做高级能力。
- AI 产品不是把信息堆满，而是减少用户判断成本。
- 资料不足时，不要假装能判断，要明确告诉用户缺什么。
- 复杂证据可以存在，但默认不打扰用户。

### 11.3 每次用户反馈都转成任务

用户反馈不要只记原话，要转成“问题 + 原因 + 任务”。

示例：

| 用户反馈 | 真实问题 | 产品任务 |
| --- | --- | --- |
| 页面不像产品化页面 | 信息结构和视觉语言不统一 | 重构主界面为输入、资料、结果、结论 |
| 结果不要小格子 | 用户要快速读结论，不要看证据堆叠 | 结果区改为三段正文 |
| 不需要像素级 diff | 验收重点是模块、入口、文案、功能 | 视觉 diff 降级为参考证据 |
| 可以丢文件夹地址 | 资料入口不应该只靠手动上传 | 增加本地资料搜索 |

### 11.4 建议建立三个文档

#### PRD

记录产品为什么做、做什么、不做什么。

#### 迭代日志

记录每次改了什么、为什么改、验证结果是什么。

#### 决策记录

记录关键分歧，例如：

- 为什么从个人知识库转成专题验收助手？
- 为什么不做像素级 diff？
- 为什么主界面只保留三段结论？
- 为什么高级证据默认折叠？

### 11.5 复盘问题清单

每次真实跑一个项目后，问自己：

1. 用户第一步是不是知道该做什么？
2. 工具有没有自动识别出关键资料？
3. 哪一步让用户等待或困惑？
4. 输出结论能不能直接用于沟通？
5. 哪些证据还需要人工补充？
6. 这次踩的坑，下次能不能变成默认规则？

## 12. 下一步建议

短期建议先做三件事：

1. 高级细节 tabs 化，让证据、OCR、点测、报告更好找。
2. 增加最终验收状态：可上线、有风险可上线、阻塞上线、资料不足。
3. 跑一个真实专题，把真实反馈记录到迭代日志里，再决定是否接入多模态 AI。
